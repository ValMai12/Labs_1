# Traceability Matrix — Online Retail System

|---|--------------|----------------|----------------|

| 9 | UC_SelectDelivery | Вибір доставки | Вибір адреси та методу доставки. |

|11 | UC_Pay | Оплата | Виконання платежу через зовнішній шлюз. |
|12 | UC_Track | Відстеження доставки | Отримання статусу замовлення від служби доставки. |
|13 | UC_Review | Залишити відгук | Оцінка товару після покупки. |
|14 | UC_Wishlist | Список бажаного | Збереження товарів для подальших покупок. |
|15 | UC_AdminUsers | Керування користувачами | Адміністрування ролей, блокування акаунтів. |

|20 | UC_MgrConfirm | Підтвердження замовлення | Затвердження нових замовлень менеджером. |
|21 | UC_MgrPrepare | Підготовка відправлення | Комплектація замовлень перед відправкою. |
|22 | UC_MgrUpdate | Оновлення статусу доставки | Синхронізація статусу від служби доставки. |
|23 | UC_SupCancel | Скасування замовлення | Скасування замовлення підтримкою. |
|24 | UC_SupRefund | Повернення коштів | Обробка запиту на повернення. |
|25 | UC_SupHelp | Підтримка користувача | Відповіді на запити клієнтів через службу підтримки. |

## 2️⃣ Functional Requirements (FR)

| FR | Функціональна вимога |
|----|----------------------|
| FR1 | Система має підтримувати реєстрацію, авторизацію та відновлення пароля. |
| FR2 | Користувач повинен мати змогу переглядати каталог і шукати товари. |
| FR3 | Повинно бути можливим додавання товарів у кошик і оформлення покупки. |
| FR4 | Підтримується інтеграція з платіжними системами для обробки транзакцій. |
| FR5 | Система має забезпечувати відстеження доставки та історію замовлень. |
| FR6 | Покупець може залишати оцінки та текстові відгуки. |
| FR7 | Можна застосовувати купони або промокоди під час оформлення замовлення. |
| FR8 | Адміністратор має керувати користувачами, товарами, акціями та відгуками. |
| FR9 | Менеджери мають опрацьовувати замовлення, готувати відправлення та оновлювати статуси. |
| FR10 | Підтримка має відповідати на запити, обробляти повернення й скасування замовлень. |
| FR11 | Система надсилає автоматичні повідомлення (email) про статуси замовлень. |
| FR12 | Користувачі можуть створювати персональні списки бажаних товарів. |
| FR13 | Система зберігає історію покупок і дій для аналітики. |

---

## 3️⃣ Non-Functional Requirements (NFR)

| NFR | Нефункціональна вимога |
|------|------------------------|
| NFR1 | Дані користувачів зберігаються у хешованому вигляді. |
| NFR2 | Середній час обробки запиту не перевищує 3 с. |
| NFR3 | Доступність системи повинна бути не нижче 99.5%. |
| NFR4 | Всі платіжні операції проходять через захищений канал (HTTPS). |
| NFR5 | Інтерфейс має бути дружнім, зрозумілим і адаптивним. |
| NFR6 | API-виклики повинні обробляти збої повторними запитами. |
| NFR7 | Архітектура має підтримувати масштабування при зростанні навантаження. |
| NFR8 | Усі транзакції повинні бути атомарними. |
